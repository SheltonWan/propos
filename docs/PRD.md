## 一、业务背景与痛点分析

本项目旨在为自有混合型物业构建一套内部数字化资产运营管理平台，将资产状态、租务合同、财务收支与物业工单四大核心业务统一纳入系统管理，实现管理层一屏掌握全局、前线员工高效协作。

### 物业资产概况

| 指标 | 数据 |
|-----|------|
| 总建筑面积（GFA）| 约 40,000 m² |
| 总房源数量 | **639 套**（写字楼 + 商铺 + 公寓）|
| A 座建面 | 约 30,000 m²（整栋写字楼）|
| 商铺建面 | 约 2,707 m² |
| 业态占比 | 写字楼 69%（约 441 套）/ 公寓 27%（约 173 套）/ 商铺 4%（约 25 套）|
| 物业管理 | 外包物业公司（Phase 1 以内部管理为主，外包公司不接入系统）|
| 现状管理方式 | 租务与财务采用传统 Excel/人工方式管理，无现有数字化系统 |
| 图纸资料 | 已有平面图及 CAD 文件（.dwg 格式）|

### 三业态核心差异对比

| 维度 | 写字楼（办公）| 商铺（零售）| 公寓（住宅）|
|-----|------------|-----------|-----------|
| 合同周期 | 通常 1~3 年 | 通常 3~5 年 | 通常 6~12 个月 |
| 租金计算 | 按建面/套内 元/m²/月 | 固定租金 ± 营业额分成 | 整套月租金 |
| 免租/装修期 | 短（1~2 周）| 长（1~3 个月）| 无 |
| 物管费标准 | 按面积独立计算 | 按面积独立计算 | 含于租金 或 租户另行缴纳（视合同约定）|
| 主要证件 | 营业执照/统一社会信用代码 | 营业执照/统一社会信用代码 | 居民身份证 |
| 水电结算 | 分表独立结算 | 分表独立结算 | 分表或含于租金 |

| 现状痛点 | 具体表现 |
|---------|---------|
| 资产信息碎片化 | 楼层平面、改造记录依赖纸质文件与 CAD 图，无统一数字资产台账 |
| 租务管理被动 | 合同到期、租金逾期无自动预警，完全靠 Excel 与人工记录 |
| 财务核算滞后 | NOI（净营运收入）无法实时查看，依赖人工月底汇总 |
| 工单流转无序 | 报修靠微信群沟通，维修成本无法归口统计 |
| 外包物业管理盲区 | 物业公司工作无数字化记录，服务质量难以量化考核（Phase 2 解决）|

## 三、Phase 1 — 功能模块详细需求

### Phase 1 范围分级与交付边界

为避免首版范围失控，Phase 1 需求按 Must / Should / Could 分级管理。项目排期、验收和资源优先保障 Must 项，不以 Could 项是否完成作为首批上线阻塞条件。

| 分级 | 定义 | 本期范围 |
|------|------|---------|
| Must | 首批上线必须完成，直接构成业务闭环 | 资产台账、CAD 基础展示、合同录入与状态机、租金递增规则、自动账单、收款核销、基础 NOI、内部工单闭环、二房东人工录入、二房东 Web 填报与审核、审计日志、组织架构管理、KPI 正式考核（含方案配置/自动冻结/申诉/排名/同比环比/Excel 导出） |
| Should | 高价值增强能力，可与 Must 并行推进，但不阻塞首批上线 | 资产概览看板、WALE 趋势、催收模板、楼栋/业态下钻报表、穿透出租率与穿透视图 |
| Could | 可预留设计与接口，不作为首批交付承诺 | 二房东 API 自动推送、复杂溢价分析、消息通道扩展、小程序增强能力 |

> **交付原则**: 若排期与资源发生冲突，优先保证主数据、合同、账单、收款、工单、二房东填报六条核心链路闭环，KPI 正式考核作为必交能力紧随其后，其余能力顺延到 Phase 1.5 或 Phase 2。

### 模块 1：资产与空间可视化

**目标**: 建立"楼栋 → 楼层 → 单元"三级数字台账底座，支持写字楼、商铺、公寓三业态的差异化字段管理。

#### 核心功能

| 功能点 | 说明 |
|--------|------|
| 楼栋档案 | 楼栋名称（A座/商铺区等）、总楼层、业态类型、GFA、NLA，支持多楼栋并行管理 |
| **CAD 平面图导入与展示** | 支持上传 .dwg 转换为 SVG/PNG 分层展示，按楼层切换浏览；系统在图层上叠加单元热区与状态色块 |
| 单元元数据档案 | 单元编号、套内面积、层高、朝向、装修状态、**业态分类（写字楼/商铺/公寓）**、改造历史、当前出租状态、**参考市场租金**（运营定期维护，用于空置损失测算与 PGI 估算） |
| 业态差异化字段 | 写字楼：工位数/分隔间数；商铺：门面宽度/临街面/层高；公寓：卧室数/独立卫生间 |
| 楼层切片状态色块 | 🟢 已租 / 🟡 即将到期（≤90天）/ 🔴 空置 / ⚪ 非可租区域，叠加于 CAD 平面图上，可按业态筛选过滤 |
| 改造记录管理 | 绑定单元，记录改造类型、日期、施工造价，支持上传改造前后照片 |
| 资产台账导出 | 全部 639 套单元数据支持按业态分类导出 Excel |
| 资产概览看板 | 按业态汇总：总套数 / 已租套数 / 空置套数 / 出租率，分写字楼、商铺、公寓三列展示 |

#### 图纸与热区管理约束

| 约束项 | 说明 |
|--------|------|
| 图纸版本 | 同一楼层允许存在多个图纸版本，默认仅展示最新生效版本，历史版本只读保留 |
| 热区绑定 | 单元热区必须与 `unit_id` 一一绑定，禁止仅以单元编号文本悬浮匹配 |
| 单元变更 | 单元拆分、合并、停租、转非可租时，需记录 `predecessor_unit_ids`（前序单元 ID 列表），旧单元标记为 `archived` 而非物理删除，历史合同仍指向旧 `unit_id`，确保历史合同与图纸不脱钩 |
| 图纸替换 | 图纸替换后不得覆盖历史审计记录，需保留上传人、上传时间和版本说明 |

> **数据初始化说明**: 639 套房源须通过 **Excel 批量导入模板**完成（按业态提供三张模板），手工逐条录入不现实。楼层平面图已有 CAD 文件，可由后端工具批量将 .dwg 转换为分层 SVG/PNG，无需人工重新绘制，大幅降低初始化工作量。

### 模块 2：租务与合同生命周期管理

**目标**: 从报价到合同终止的全流程数字化，消灭信息死角。

#### 2.1 租客全景画像（Tenant Profile）

| 数据维度 | 说明 |
|---------|------|
| 基本信息 | 企业/个人、统一社会信用代码/身份证、联系人、紧急联系方式 |
| 租赁历史 | 当前及历史合同、曾租单元、续租次数 |
| 缴费信用 | 历史逾期次数、最近付款日，系统自动生成信用评级（A/B/C），评级规则见下表 |

##### 租户信用评级量化规则

| 评级 | 规则 |
|------|------|
| **A（优质）** | 过去 12 个月逾期 ≤ 1 次，且单次逾期天数 ≤ 3 天 |
| **B（一般）** | 过去 12 个月逾期 2~3 次，或单次逾期 4~15 天 |
| **C（风险）** | 过去 12 个月逾期 ≥ 4 次，或单次逾期 > 15 天 |

> 信用评级每月 1 日自动重算；新租户默认为 B 级，签约满 3 个月后首次评级。
| 工单记录 | 关联该租客历史报修记录 |

#### 2.2 合同管理

| 功能点 | 说明 |
|--------|------|
| 合同录入 | **多单元绑定**（一份合同可关联多个单元，通过 `contract_units` 中间表记录每个单元的计费面积与单价分摊）、起租日、到期日、月租金、含税/不含税标识、免租/装修期、押金、付款周期、**业态自动关联** |
| 附件管理 | 上传合同 PDF 及补充协议 |
| 合同状态机 | 报价中 → 待签约 → 执行中 → 即将到期 → 已到期/已续签/已终止；执行中可直接转为已终止（提前终止路径），详见 2.2.2 |
| 续签管理 | 续签合同与原合同关联，形成完整合同链 |
| 免租/装修期处理 | 免租期（写字楼通常 1~2 周，商铺通常 1~3 个月）内账单自动标记，不计入逾期 |
| 商铺营业额分成（可选）| 商铺合同支持录入“保底租金 + 营业额分成”计费规则，其中当月应收取保底租金与营业额分成额两者较高值；商户按月申报营业额 → 财务审核通过 → 系统自动计算分成账单（流程类似二房东提交-审核机制，详见 2.2.3） |

#### 2.2.1 计费与核销规则矩阵

| 规则项 | 统一口径 |
|--------|----------|
| 租金计费基准 | 写字楼、商铺默认按合同约定计费面积 × 单价；公寓默认按整套固定月租。合同必须显式记录计费面积、计费单价和税费口径 |
| 首期/末期不足月 | 默认按自然日折算，计算公式为：当期应收 = 月标准金额 ÷ 当月自然日 × 实际计费天数 |
| 免租/装修期 | 允许跨账期配置，系统生成账单时标记为免租或免单，不计逾期；若仅免租不免管理费，需按费项分别配置 |
| 押金 | 押金单独建账，不计入 NOI 收入；支持收取、冻结、部分冲抵、退还四种状态流转，详见 2.2.4 押金管理 |
| 收款核销 | 支持部分收款、一次收多账单、单账单多次核销；默认按“先到期先核销”分配，允许财务手工调整 |
| 发票管理 | 发票状态与收款状态解耦，允许“已收款未开票”“已开票未收款”并存 |
| 商铺营业额分成 | 商户按月申报营业额，财务可复核后生成分成账单；补报或修正申报时，系统需重算对应期间应收 |
| 税费处理 | 合同必须标注含税/不含税；账单同时展示含税金额与不含税金额；NOI 计算统一使用**不含税口径**。物业租赁涉及增值税（一般纳税人 9%、简易征收 5%）、房产税（从租 12%）等，系统需记录适用税率供账单计算 |
| NOI 管理口径 | Phase 1 默认展示**经营管理口径**（按合同实际应收计），不做会计准则调整（如免租期直线法分摊），并提供应收、实收两个视角；押金、代收代付不计入 NOI。Phase 2 可扩展为双口径（经营 + 会计） |

#### 2.2.2 合同提前终止规则

实务中高频出现租户提前退租或双方协商终止，需在合同状态机中显式支持：

| 终止类型 | 说明 |
|---------|------|
| **正常到期** | 合同自然到期，走标准状态机路径 |
| **租户提前退租** | 租户主动违约退租，需记录退租日期、违约金金额、押金扣除明细 |
| **协商提前终止** | 双方协商一致提前解约，需记录协商条款与补偿方案 |
| **业主单方解约** | 租户严重违约（如连续逾期超过约定天数），业主方解约，需记录解约依据 |

> **关联影响**: 提前终止时，系统需自动终止后续未生成账单、关闭递增规则、触发押金退还/扣除流程，并将合同终止原因写入审计日志。WALE 计算中该合同剩余租期归零。

#### 2.2.3 商铺营业额分成对账机制

| 流程节点 | 说明 |
|---------|------|
| 商户申报 | 商户按月提交营业额数据（支持手工录入或上传 POS 汇总） |
| 财务审核 | 财务人员审核营业额数据真实性，可要求商户补充证明材料 |
| 生成分成账单 | 审核通过后，系统按「取 **保底租金** 与 **营业额 × 分成比例** 两者中较高值」自动生成当月应收账单（即 `MAX(保底租金, 营业额 × 分成比例)`） |
| 补报/修正 | 商户补报或修正营业额时，系统需重算对应期间应收并生成差额账单 |
| 争议处理 | 发生争议时需记录争议内容与处理结果，保留审计记录 |

> **业主抽查权**: 合同条款中应约定业主方有权抽查商户 POS 流水或经审计的财务报表，系统可记录抽查记录与结论。

#### 2.2.4 押金管理

押金在物业管理中是高频纠纷点，需独立于租金账单体系单独管理：

| 功能点 | 说明 |
|--------|------|
| 押金独立建账 | 押金不计入 NOI 收入，独立记录金额、收取日期、关联合同 |
| 状态流转 | `已收取 → 冻结中 → 部分冲抵 → 已退还`，每次状态变更需记录原因和审批人 |
| 违约扣除 | 合同提前终止或租户违约时，支持从押金中扣除违约金/欠费，并生成扣除明细 |
| 转续签合同 | 续签时押金可整体转移至新合同，无需先退后收 |
| 退还审批 | 押金退还需经财务确认无欠费后审批，退还金额、方式（原路退回/转账）均需记录 |

> 押金每次变更（收取、冻结、冲抵、退还）均写入审计日志，保留完整流水记录。

#### 2.3 智能预警引擎（Alert Engine）

| 预警类型 | 触发逻辑 | 接收方 |
|---------|---------|--------|
| 租约到期预警 | 提前 90 / 60 / 30 天触发 | 租务专员 + 运营管理层 |
| 租金逾期预警 | 账期截止后第 1 / 7 / 15 天未到账 | 财务 + 租务专员 |
| 月度到期汇总 | 每月 1 日自动推送 | 管理层 |
| 押金退还提醒 | 合同终止前 7 天 | 财务人员 |

> **通知渠道**: 系统内消息中心 + 邮件；后期可扩展企业微信/飞书 Webhook。

#### 2.3.1 预警执行与补偿规则

| 场景 | 规则 |
|------|------|
| 触发频率 | 同一合同同一类预警在同一自然日仅触发一次，避免重复轰炸 |
| 发送失败 | 邮件或站内消息发送失败时，任务需自动重试至少 3 次，并记录失败原因 |
| 人工补发 | 管理后台支持按合同或按日期区间手工补发预警 |
| 巡检要求 | 运营方需每日查看失败任务列表，确保提醒任务可追溯 |

#### 2.4 动态 WALE 计算（加权平均租期到期年数）

WALE（Weighted Average Lease Expiry）是商业地产资产管理核心指标，反映租约稳定性与估值基础。参照 MSCI/IPD 行业标准，系统同时提供**收入加权 WALE** 和 **面积加权 WALE** 两个口径：

$$WALE_{\text{收入}} = \frac{\sum (剩余租期_i \times 年化合同租金_i)}{\sum 年化合同租金_i}$$

$$WALE_{\text{面积}} = \frac{\sum (剩余租期_i \times 计费面积_i)}{\sum 计费面积_i}$$

> **精度要求**: $剩余租期_i$ 计算精确到天（而非到月），单位为年（如 2.35 年）。多单元合同按单元拆分后分别计入 WALE，避免同一合同被重复加权。

| 展示维度 | 说明 |
|---------|------|
| 组合级 WALE | 全部在租合同加权平均到期年数（收入加权 + 面积加权并列展示） |
| 楼栋级 WALE | 按楼栋独立计算（A座写字楼、商铺区、公寓楼） |
| **业态级 WALE** | 写字楼 / 商铺 / 公寓分别独立呈现（三业态租约稳定性差异显著）|
| WALE 趋势图 | 过去 12 个月变化曲线 |
| 到期瀑布图 | 按年份显示未来租约到期面积分布 |

#### 2.5 租金递增规则配置器

商业地产的租赁合同中，租金递增方式因业态、租户谈判结果不同而千差万别。本配置器支持**灵活选择、组合拼接、模板复用**，并与账单自动生成和 WALE 计算深度联动。

##### 支持的递增类型

| 递增类型 | 说明 | 示例 |
|---------|------|------|
| **固定比例递增** | 每个递增周期按固定百分比上涨 | 每年涨 5% |
| **固定金额递增** | 每个递增周期按固定单价上涨 | 每年涨 ¥3/m² |
| **阶梯式递增** | 按合同年限分段设定不同租金 | 第 1~2 年 ¥80/m²，第 3~4 年 ¥90/m²，第 5 年 ¥100/m² |
| **CPI 挂钩递增** | 每年按上年度 CPI 涨幅调整（系统支持手动录入年度 CPI 或对接统计局公开数据）| 上年 CPI 2.3%，本年租金涨 2.3% |
| **每 N 年递增** | 指定递增间隔年数与涨幅 | 每 2 年涨一次 8% |
| **免租后基准调整** | 免租/装修期结束后，首年按约定基准价起算，后续可叠加其它递增规则 | 免租 3 个月后，首年 ¥75/m² 起步 |

##### 混合递增（分段组合）

一份合同可设置**多个递增阶段**，每个阶段独立配置递增类型。系统按时间线串联执行：

```
合同总期 5 年：
├── 阶段 1（第 1~2 年）：固定租金 ¥80/m²（免租后基准）
├── 阶段 2（第 3~4 年）：每年固定比例递增 5%
└── 阶段 3（第 5 年起）：CPI 挂钩递增
```

| 配置项 | 说明 |
|-------|------|
| 阶段起止 | 以合同起租日为基准，按年或月设定每个阶段的起止时间 |
| 递增类型选择 | 下拉选择上述 6 种递增类型之一 |
| 参数录入 | 根据选定类型动态展示对应参数输入框（百分比 / 金额 / 阶梯表 / CPI 年份录入等）|
| 阶段数量 | 不限，支持增减阶段 |

##### 递增模板管理

| 功能点 | 说明 |
|--------|------|
| 创建模板 | 将配置好的递增规则保存为命名模板（如"写字楼标准5年递增"、"商铺保底+阶梯"）|
| 模板套用 | 新合同录入时可一键选用已有模板，参数自动填入，仍可微调 |
| 模板列表 | 按业态分类展示所有模板，支持搜索、编辑、删除 |
| 默认模板 | 可为每个业态设定默认递增模板，新合同创建时自动带入 |

##### 自动计算联动

| 联动模块 | 说明 |
|---------|------|
| **账单自动生成（3.1）** | 系统根据递增规则自动计算每期应收租金，账单金额随递增周期自动调整，无需人工手动修改 |
| **WALE 计算（2.4）** | WALE 公式中的「年化合同租金」取递增后实际租金而非签约基准租金，确保 WALE 反映真实租金加权 |
| **未来租金预测** | 根据递增规则自动生成合同全生命周期的租金预测表（按月/年展示），支持导出 Excel |
| **续签对比** | 续签时自动对比原合同末期租金与新合同起始租金，计算实际涨跌幅 |

### 模块 3：财务与业财一体化

#### 3.1 账单自动生成与催收

|--------|------|
| 自动账单生成 | 依据合同条款，在每个账期前自动生成应收账单 |

#### 3.1.1 水电分表结算方案

| 规则项 | 说明 |
|--------|------|
| 抄表周期 | 默认月度抄表；支持配置双月抄表周期 |
| 抄表方式 | Phase 1 采用手工录入抄表读数（起止度数），远传智能电表对接留 Phase 2 |
| 计费规则 | 按「本期读数 - 上期读数 = 用量 × 单价」计算；单价支持按业态独立配置 |
| 阶梯水电价 | 支持配置阶梯式水价/电价（如用量 ≤ X 度时单价 A，超过部分单价 B） |
| 公摊分摊 | 公共区域水电费按面积比例分摊至各租户（分摊比例 = 租户计费面积 ÷ 总计费面积） |
| 业态差异 | 写字楼/商铺：独立分表结算；公寓：视合同约定，可含于租金或独立结算 |
| 账单生成 | 录入抄表数据后自动生成水电费账单，附带用量明细 |

#### 3.2 净营运收入（NOI）实时看板

$$NOI = 有效总收入\ (EGI) - 运营支出\ (OpEx)$$

$$EGI = 潜在总收入\ (PGI) - 空置损失 + 其他收入$$

> **口径声明**: Phase 1 NOI 为**经营管理口径**（按合同实际应收计），不做会计准则调整（如免租期直线法分摊、递增租金实际利率法确认等）。本看板不代替合规财务报表，Phase 2 可扩展为双口径（经营 + 会计）。

##### NOI 三级展示架构

NOI 信息量大且受众不同，采用**「概览 → 看板 → 明细」**三级渐进披露：

| 层级 | 位置 | 展示内容 | 优先级 |
|------|------|---------|--------|
| **L1 概览** | 首页 Dashboard 指标卡片 | 当月 NOI（金额 + 环比变化箭头）、NOI Margin、出租率、收款率——4~5 个浓缩数字，点击跳转 L2 | Must |
| **L2 看板** | 财务模块 → NOI 看板（独立页面） | 下方看板组件表格的全部内容：业态分拆、支出类目、趋势图、楼栋下钻、预算对比等 | Must + Should |
| **L3 明细** | L2 看板内下钻（弹窗 / 子页面） | 具体楼栋 NOI 逐笔明细、支出清单、未缴款租户列表、空置单元列表 | Should |

> **设计原则**：管理层打开系统第一件事是看经营健康度（L1），需要分析时进入完整看板（L2），追因时下钻到明细（L3）。uni-app 移动端仅展示 L1，L2/L3 通过 Admin PC 端承载。

| 看板组件 | 说明 |
|---------|------|
| 月度 NOI 卡片 | 当月实收租金 / 运营支出 / NOI 三栏并列实时展示 |
| **业态 NOI 分拆** | 写字楼 / 商铺 / 公寓三业态分别展示收入、支出与 NOI，便于横向比较回报率 |
| 运营支出类目 | 水电公摊、保洁保安（外包物业费）、维修费、保险、税金、专业服务费（消防检测/电梯年检等） |
| **NOI Margin & OpEx Ratio** | NOI Margin = NOI ÷ EGI × 100%、OpEx Ratio = OpEx ÷ EGI × 100%，与 NOI 卡片并列展示，无额外取数成本 |
| 出租率 | 已租 NLA ÷ 总可租 NLA，可按业态下钻（写字楼出租率 / 公寓出租率 / 商铺出租率分列）|
| 空置损失测算 | 空置单元的市值租金损失估算（基于单元"参考市场租金"字段，由运营定期维护） |
| 收款进度 | 本月应收 vs 实收对比（639 套规模下需支持快速检索未缴款租户）|
| 历史趋势 | 过去 12 个月 NOI 折线图 |
| 楼栋下钻 | 管理层可按楼栋（A座/商铺区/公寓楼）独立查看 NOI |
| **NOI 年度预算** | 支持按楼栋/业态录入年度 NOI 预算（Excel 导入或手动录入），作为 K07 NOI 达成率的对比基准 |

#### 3.3 自定义 KPI 仪表盘

为运营管理层提供**可配置权重**的量化分析工具，自动从系统实时数据中提取指标，对员工/部门进行 KPI 评分与排名。

> **Phase 1 定位说明**: KPI 模块在 Phase 1 作为正式考核工具，评分结果为独立考核报告，不对接薪酬系统。组织架构通过 `departments` 三级组织树建模（公司→部门→组），员工管辖范围支持部门默认 + 个人覆盖双机制，KPI 数据归集按管辖范围自动筛选。

##### 组织架构前置（KPI 正式考核依赖）

KPI 正式考核需要明确的组织归属与管辖范围，系统通过以下机制支撑：

| 配置项 | 说明 |
|-------|------|
| 三级组织树 | 公司 → 部门 → 组（通过 `departments` 表 `parent_id` 自引用实现，最大 3 级） |
| 员工归属 | 每个员工绑定一个部门（`users.department_id`），一个员工仅属于一个部门 |
| 管辖范围 | 部门可配置默认管辖的楼栋/楼层/业态；个人可覆盖部门默认范围 |
| 数据归集 | KPI 指标计算时，按评估对象的最终管辖范围（个人覆盖 > 部门默认）筛选原始数据 |

##### 评估对象

| 对象类型 | 说明 |
|---------|------|
| **员工** | 按员工关联的管辖楼栋/楼层/业态范围自动归集指标数据 |
| **部门** | 租务部、财务部、物业运营部等，汇总部门下所有员工数据 |

##### 系统预定义指标库

以下指标由系统从实时运营数据中自动计算，管理员只需配置权重和满分阈值：

| 指标编号 | 指标名称 | 数据来源 | 默认满分标准 |
|---------|---------|---------|------------|
| K01 | **出租率** | 模块 1 资产台账（已租 NLA ÷ 总可租 NLA） | ≥95% 满分 |
| K02 | **收款及时率** | 模块 3 账单核销（账期内到账笔数 ÷ 应收笔数） | ≥95% 满分 |
| K03 | **租户集中度** | 模块 2 合同数据（前 3 大租户租金占比） | ≤40% 满分 |
| K04 | **续约率** | 模块 2 合同状态机（续签合同数 ÷ 到期合同数） | ≥80% 满分 |
| K05 | **工单响应时效** | 模块 4 工单系统（仅 `repair` 类型：平均派单到完工时长） | ≤24h 满分 |
| K06 | **空置周转天数** | 模块 1 + 模块 2（单元空置至重新签约的平均天数） | ≤30 天满分 |
| K07 | **NOI 达成率** | 模块 3 NOI 看板（实际 NOI ÷ 预算 NOI） | ≥100% 满分 |
| K08 | **逾期率** | 模块 3 账单（逾期金额 ÷ 应收金额） | ≤5% 满分 |
| K09 | **租金递增执行率** | 模块 2 递增配置器（按递增规则成功调价的合同数 ÷ 应调价合同数） | ≥95% 满分 |
| K10 | **租户满意度**（手动录入） | 管理员按季度手动录入调查评分 | ≥90 分满分 |

> **扩展性**: 指标库为系统预定义，管理员可对每个指标「启用/停用」，不需要的指标可关闭；未来版本可支持管理员自定义计算公式新增指标。

> **指标方向说明**: K03（租户集中度）、K05（工单响应时效）、K06（空置周转天数）、K08（逾期率）为**反向指标**（数值越低越好）；其余为正向指标。反向指标的线性插值逻辑需翻转：实际值 ≤ 满分标准时得 100 分，实际值 ≥ 及格标准时得 60 分，超出及格标准按比例递减至 0 分。

##### KPI 方案配置

支持**多套 KPI 方案并存**，不同部门/岗位可应用不同方案：

| 配置项 | 说明 |
|-------|------|
| 方案名称 | 如"租务部考核方案 2026"、"物业运营部季度考核" |
| 指标选择 | 从预定义指标库中勾选本方案使用的指标（不必全选） |
| 权重分配 | 为每个启用指标分配权重（百分比），所有权重之和必须 = 100% |
| 满分阈值调整 | 可针对本方案微调每个指标的满分/及格/不及格阈值 |
| 评估周期 | 月度 / 季度 / 年度（决定取数时间范围） |
| 适用对象 | 绑定到具体部门或员工 |
| 生效时间 | 方案的起止有效期，支持版本迭代（新方案替代旧方案时旧数据保留） |

> **数据冻结规则**: 每个评估周期结束后生成评分快照。周期关闭后默认不自动回算，若发生补录或口径修订，需由有权限角色发起重算并保留重算审计记录。

##### 打分计算逻辑

$$KPI_{总分} = \sum_{i=1}^{n} (指标得分_i \times 权重_i)$$

每个指标得分按线性插值计算：

| 指标实际值 vs 阈值 | 得分 |
|-------------------|------|
| ≥ 满分标准 | 100 分 |
| 满分标准 ~ 及格标准之间 | 线性插值（60~100 分） |
| < 及格标准 | 按比例递减（0~60 分） |

##### 展示与历史对比

| 功能点 | 说明 |
|--------|------|
| KPI 总览看板 | 按部门/员工展示当期 KPI 总分、各指标得分雷达图 |
| 排名榜 | 同一 KPI 方案下的员工/部门排名，支持按指标筛选 |
| 历史趋势 | 员工/部门过去 6~12 个月的 KPI 得分折线图，识别改善或下滑趋势 |
| 同比/环比 | 与上月、去年同期 KPI 得分对比 |
| 指标下钻 | 点击某个指标得分可下钻查看原始数据明细（如哪些合同逾期、哪些单元空置） |
| 导出报告 | KPI 评分结果支持导出 Excel，供管理层会议使用（Phase 1 先支持 Excel，PDF 后续扩展） |

##### 申诉机制

评分快照冻结后，员工可在规定时间内提交申诉：

| 配置项 | 说明 |
|-------|------|
| 申诉窗口 | 快照冻结后 **7 个自然日**内可提交申诉 |
| 申诉流程 | 员工提交申诉理由 → 运营管理层/超级管理员审核 → 批准后触发重算 |
| 重算规则 | 批准申诉后以当前数据重新计算该快照，快照状态变为 `recalculated`，保留原始快照用于审计 |
| 审计要求 | 申诉提交、审核决定、重算触发全程写入审计日志 |

| 操作 | 可执行角色 |

### 模块 4：物业运营与工单系统

**目标**: 物业运营事务线上化，覆盖报修、投诉、退租验房三类工单，维修成本自动汇入 NOI 支出端。

#### 4.0 工单类型定义

Phase 1 支持以下三种工单类型，共享同一张 `work_orders` 表，通过 `work_order_type` 枚举区分：

| 枚举值 | 中文名 | 适用场景 | 说明 |
|--------|--------|---------|------|
| `repair` | **报修** | 设备故障、设施损坏等需要维修的事项 | 状态机包含完整的派单→处理→验收流程，完工后录入维修成本 |
| `complaint` | **投诉** | 租客对物业服务、环境噪音、公区卫生等事项的投诉 | 不涉及维修成本，以「服务处理结论」替代「验收」环节 |
| `inspection` | **退租验房** | 合同到期或提前退租时的单元查验 | 关联合同，查验结果影响押金扣减决策，可产生维修跟进工单 |

> **Phase 2 预留枚举值**（当前不实现）：`patrol`（巡检）、`cleaning`（保洁）、`maintenance`（设备保养）、`security`（安防事件）、`renovation_request`（装修申请）。

**各类型流程差异**:

| 流程节点 | `repair` 报修 | `complaint` 投诉 | `inspection` 退租验房 |
|---------|--------------|-----------------|---------------------|
| 提报 | 选择位置 + 问题类型 + 照片 | 选择位置 + 投诉类型 + 描述 | 选择合同 + 单元（自动关联） |
| 审核/派单 | 派给维修人员/供应商 | 派给客服/主管 | 派给验房员 |
| 处理中 | 维修处理 | 服务处理 | 逐项查验并拍照记录 |
| 完工/处理完毕 | 录入材料费 + 人工费 + **费用性质**（经常性 OpEx / 资本性 CapEx） | 录入处理结论 | 录入查验结论 + 需维修项 |
| 验收 | 现场验收 | 回访确认（可选） | 业主确认 + 押金扣减建议 |
| 成本 | 标记为 OpEx 的 ✅ 汇入 NOI；标记为 CapEx 的不计入 NOI（行业惯例：单次 > 5,000 元且延长资产使用寿命的列 CapEx） | ❌ 无成本 | ❌ 查验本身无成本（产生的维修工单有成本） |

**`issue_type`（问题子类型）按 `work_order_type` 分组**:

| `work_order_type` | `issue_type` 可选值 |
|-------------------|-------------------|
| `repair` | `水电`、`空调`、`门窗`、`消防`、`网络`、`电梯`、`其他` |
| `complaint` | `服务态度`、`环境噪音`、`公区卫生`、`安全隐患`、`其他` |
| `inspection` | `合同到期验房`、`提前退租验房` |

### 模块 5：二房东租赁信息穿透管理

**目标**: 对于大面积整租给二房东（转租方）的单元，穿透主合同层级，掌握二房东实际转租给终端租客的真实租赁信息（租客、面积、租金、入住状态），实现"主合同 → 子租赁"两级租赁数据的完整可视化。数据采集支持**内部人工录入**与**二房东外部自助录入**双通道。

#### 5.1 业务场景说明

当前两边大面积物业（写字楼/公寓）分别由一家二房东公司整租后转租。公司作为业主方，仅掌握与二房东签订的主合同信息，对终端实际租客、实际租金、空置情况完全不可见，存在以下风险：

| 痛点 | 说明 |
|------|------|
| 租赁数据不透明 | 不掌握实际入住租客身份、入住面积与真实租金水平 |
| 资产回报评估失真 | 无法判断二房东转租溢价空间，不利于续租谈判定价 |
| 空置风险不可控 | 二房东名下实际空置率不可见，影响整体资产运营决策 |
| 安全与合规隐患 | 不清楚实际入住人员信息，无法配合消防、治安等合规要求 |

#### 5.2 数据模型：主合同 → 子租赁两级关联

```
┌────────────────────────┐
│  主合同（Master Lease） │  ← 公司与二房东签订
│  二房东A：A座 5F~10F    │
└──────────┬─────────────┘
           │  1 : N
           ▼
┌────────────────────────┐
│  子租赁（Sub-Lease）    │  ← 二房东与终端租客签订
│  5F-501: 张三科技 ¥80/m²│
│  5F-502: 李四传媒 ¥85/m²│
│  5F-503: 空置           │
│  ...                    │
└────────────────────────┘
```

#### 5.3 子租赁信息数据字段

| 字段 | 说明 | 必填 |
|------|------|------|
| 所属主合同 | 关联二房东主合同编号 | 是 |
| 单元编号 | 从主合同覆盖单元范围中选择 | 是 |
| 终端租客名称 | 企业名称或个人姓名 | 是 |
| 租客类型 | 企业 / 个人 | 是 |
| 联系人 & 联系电话 | 终端租客主要联系方式 | 是 |
| 营业执照/身份证号 | 按租客类型录入对应证件 | 否（建议填写） |
| 子租赁起止日期 | 子合同实际起租日 ~ 到期日 | 是 |
| 实际月租金 | 终端租客支付给二房东的月租金 | 是 |
| 租金单价 | 元/m²/月，系统可按面积自动反算 | 自动计算 |
| 入住状态 | 已入住 / 已签约未入住 / 已退租 / 空置 | 是 |
| 入住人数（公寓适用） | 实际入住人数 | 否 |
| 备注 | 自由文本 | 否 |

#### 5.4 双通道数据采集

##### 通道一：内部人工录入

| 功能点 | 说明 |
|--------|------|
| 手工逐条录入 | 租务专员在管理后台按主合同选择单元，逐条录入子租赁信息 |
| Excel 批量导入 | 提供子租赁信息导入模板，支持按二房东批量导入 |
| 数据校验 | 单元编号必须在主合同覆盖范围内；子租赁到期日不得超出主合同到期日；同一单元不可重复录入在租记录 |

##### 通道二：二房东外部自助录入接口

| 功能点 | 说明 |
|--------|------|
| 独立登录入口 | 为每家二房东分配独立账号密码，通过 **Web 端轻量表单页面**登录（非管理后台，独立前端页面） |
| 数据权限隔离 | 二房东仅可查看/编辑其自身主合同范围内的单元与子租赁信息，不可见其他任何系统数据 |
| 子租赁信息填报 | 二房东登录后看到其名下所有单元列表，可对每个单元填写/更新终端租客信息 |
| Excel 批量上传 | 二房东可下载标准模板，填写后批量上传子租赁数据 |
| 变更记录留痕 | 二房东每次提交/修改均记录操作时间、修改前后内容、操作人，方便审计 |
| 数据审核机制 | 二房东提交后状态为"待审核"，租务专员/运营管理层审核通过后方生效；审核不通过可附拒绝原因退回 |
| 提交提醒 | 每月固定时间（如每月 5 日）自动向二房东发送填报提醒（邮件/短信） |
| API 接口（预留） | 预留 RESTful API 接口文档，便于二房东方如有自有系统可通过 API 自动推送数据（Phase 1 提供接口定义，视二房东技术能力决定是否启用） |

#### 5.4.1 审核 SLA 与版本规则

| 场景 | 规则 |
|------|------|
| 提报截止日 | 默认每月 5 日前完成上月数据填报 |
| 审核 SLA | 租务专员或运营管理层需在 2 个工作日内完成审核 |
| 退回重提 | 被退回的数据允许重提，系统需保留每次提交版本与差异 |
| 超时升级 | 超过审核 SLA 未处理时，自动提醒管理层 |
| 生效时点 | 仅“审核通过”版本参与穿透看板、出租率与分析口径计算 |

#### 5.5 穿透分析看板

| 看板组件 | 说明 |
|---------|------|
| 二房东总览卡片 | 展示每家二房东：主合同租金 / 已填报子租赁数 / 终端实际出租面积 / 终端空置面积 |
| 转租溢价分析 | 终端平均租金单价 vs 主合同租金单价，计算二房东溢价率 = (终端均价 - 主合同单价) / 主合同单价 × 100% |
| 穿透出租率 | 二房东名下单元的终端实际出租率（非仅看主合同是否在租），可与整体出租率对比 |
| 子租赁到期预警 | 终端租客合同集中到期预警，辅助评估二房东续约稳定性 |
| 填报完整度监控 | 监控二房东数据填报覆盖率（已填报单元数 / 主合同覆盖总单元数），识别漏报 |
| 楼层穿透视图 | 在已有楼层色块图上叠加子租赁信息：鼠标悬停单元可展示终端租客名称、租金、入住状态（而非仅显示"二房东A"） |

#### 5.6 安全与权限设计

| 安全要求 | 说明 |
|---------|------|
| 账号管理 | 由超级管理员创建二房东账号，绑定主合同，账号有效期与主合同到期日联动 |
| 数据隔离 | 二房东之间严格数据隔离，不可跨合同查看 |
| 操作审计 | 二房东端所有操作（登录、查看、新增、修改、删除）均记录审计日志 |
| 接口安全 | API 接口采用 Token 鉴权 + IP 白名单（可选），防止未授权访问；**强制 HTTPS（TLS 1.2+），禁止 HTTP 明文访问** |
| 数据脱敏 | 管理后台查看终端租客证件号时默认脱敏展示，需二次授权方可查看完整信息 |
| 主合同到期自动冻结 | 主合同到期后，对应二房东账号自动冻结，数据保留但不可修改 |
| 登录安全 | 首次登录强制改密；连续 5 次登录失败锁定账号 30 分钟；**密码复杂度要求：最少 8 位，必须含大小写字母 + 数字，禁止与用户名相同** |
| 会话控制 | 外部门户默认 30 分钟无操作失效；同账号再次登录可使旧会话失效 |
| 导出控制 | 二房东默认无批量导出权限；内部导出需记录操作人、时间和导出范围 |
| 数据责任 | 二房东提交数据前需勾选真实性声明，确认其对填报内容合法性和准确性负责 |
| 留存周期 | 审计日志、审批记录、版本记录至少保留 3 年 |

## 五、非功能性要求

| 类别 | 要求 |
|-----|------|
| **访问端** | Vue3 + Element Plus Web 管理后台（PC 优先）+ uni-app 移动端 App 包（iOS/Android/HarmonyOS；工单全流程 / uni-push 通知 / CAD 快查）+ uni-app 微信小程序精简版（仅扫码报修 + 状态查看，不承担推送）|
| **性能** | 仪表盘加载 < 3 秒；支持 **50 并发用户**（Phase 1 仅内部人员使用）；账单批量生成（最大批次 639 条）< 30 秒 |
| **数据安全** | 租客证件号、账户信息加密存储；角色权限严格隔离 |
| **个人信息保护（PIPL 合规）** | 系统存储租户身份证号、手机号等个人信息，需满足《个人信息保护法》要求：① 收集前在合同条款中告知并获得授权；② 遵循最小必要原则，仅收集业务必需字段；③ 脱敏还原操作需记录审计日志（谁在什么时间查看了哪位租户的完整个人信息）；④ 二房东填报终端租客信息时，需明确数据处理授权；⑤ 合同终止后个人信息保留不超过 3 年（需符合税务留存要求） |
| **审计日志** | 合同修改、账单核销、权限变更均留操作日志 |
| **数据备份** | 每日自动备份，支持全量数据导出 |
| **可扩展性** | 系统架构支持 Phase 2 模块平滑接入 |
| **恢复目标** | 生产环境建议 RPO ≤ 24 小时，RTO ≤ 4 小时 |
| **任务可靠性** | 账单生成、预警推送、提醒发送等定时任务失败后需自动重试，并提供失败任务列表 |
| **可观测性** | 关键接口、批量导入、任务调度、文件转换需有结构化日志与错误告警 |
| **留存策略** | 审计日志、账单、合同附件、导入记录需定义留存周期与归档策略 |
| **语言** | Phase 1 仅支持**简体中文**，不做国际化 |
| **离线模式** | Phase 1 uni-app 移动端仅支持**在线模式**，离线缓存与同步能力留 Phase 2（外包物业巡检场景更迫切） |

## 附录 A：WALE 行业参考资料

> 本附录为内部参考，帮助团队理解 WALE 指标在商业地产行业中的定位与解读方式，不构成需求规格。

### A.1 指标定义与行业地位

WALE（Weighted Average Lease Expiry，加权平均租期到期年数）是商业地产和 REITs（房地产投资信托基金）行业最核心的租约稳定性指标之一，反映**现有租约组合距离到期的加权平均剩余时间**。WALE 越高，表明租金收入的确定性和持续性越强。

该指标被 **MSCI/IPD**（全球不动产指数）、**APREA**（亚太房地产协会）、各国 REITs 监管机构列为必须披露的关键运营指标。

### A.2 两种标准口径

业界通用两种加权方式，本系统（M2.4）均已实现：

**收入加权 WALE（Income-weighted）**

$$WALE_{\text{收入}} = \frac{\sum_{i=1}^{n} (T_i \times R_i)}{\sum_{i=1}^{n} R_i}$$

- $T_i$ = 第 $i$ 份租约的剩余租期（年），精确到天（$T = \frac{\text{剩余天数}}{365.25}$）
- $R_i$ = 第 $i$ 份租约的年化合同租金（取递增后当前实际租金）

**侧重点**：高租金租户权重更大，适合评估**收入稳定性和现金流风险**。

**面积加权 WALE（Area-weighted）**

$$WALE_{\text{面积}} = \frac{\sum_{i=1}^{n} (T_i \times A_i)}{\sum_{i=1}^{n} A_i}$$

- $A_i$ = 第 $i$ 份租约的计费面积

**侧重点**：大面积租户权重更大，适合评估**空间利用率和再招租压力**。

> 两个口径可能产生显著差异。例如一个高单价小面积租约到期，对收入加权 WALE 的冲击大于面积加权 WALE。

### A.3 计算精度行业惯例

| 要素 | 行业惯例 |
|------|---------|
| 剩余租期 $T_i$ | 精确到**天**，以年为单位（用 365.25 考虑闰年） |
| 截止时点 | 以**计算日**（通常为报告期末或当日）为起点 |
| 多单元合同 | 按单元拆分后分别计入，避免同一合同被重复加权 |
| 纳入范围 | 仅**在租/执行中**合同；已终止、草稿、待生效合同不参与 |
| 租金取值 | 取**当前递增后实际租金**，而非签约时基准租金 |

### A.4 行业基准值

| 物业类型 | 健康 WALE 范围 | 说明 |
|---------|--------------|------|
| 甲级写字楼 | 3.0 ~ 5.0 年 | 企业租户倾向签长约 |
| 零售/商铺 | 4.0 ~ 7.0 年 | 品牌连锁通常签 5 年以上 |
| 公寓/住宅 | 0.5 ~ 1.0 年 | 短租为主，WALE 天然偏低 |
| 混合业态组合 | 2.0 ~ 4.0 年 | 视业态结构而定 |
| REITs 披露中位数 | 3.5 ~ 5.0 年 | 新加坡/香港上市 REITs 公开报告 |

> 本项目三业态混合（写字楼 69% + 公寓 27% + 商铺 4%），公寓占比较高会拉低整体 WALE，预估健康范围约 **2.0 ~ 3.5 年**。

### A.5 业务解读维度

| 分析角度 | 说明 |
|---------|------|
| **估值基础** | 机构投资者和银行评估物业价值时，WALE 是核心参考指标。WALE 越高，DCF 估值中的现金流确定性越强 |
| **再融资风险** | WALE < 2 年通常被视为黄色预警，短期内大量租约需续签，存在空置风险 |
| **集中到期风险** | 单看 WALE 均值可能掩盖问题——需配合**到期瀑布图**（Lease Expiry Profile）识别某一年是否集中到期 |
| **业态分拆** | 三业态独立计算 WALE 非常重要：公寓 WALE 0.8 年是正常的，但写字楼 WALE 0.8 年则是严重警告 |
| **趋势监控** | WALE 持续下降意味着续签速度跟不上到期速度，需提前启动招商 |

## 附录 B：NOI 行业参考资料

> 本附录为内部参考，帮助团队理解 NOI 指标在商业地产行业中的定位与解读方式，不构成需求规格。

### B.1 指标定义与行业地位

NOI（Net Operating Income，净营运收入）是商业地产领域最核心的盈利能力指标，衡量物业在日常运营中产生的**净现金流能力**。它被 NCREIF（美国全国房地产投资信托委员会）、MSCI/IPD、RICS（皇家特许测量师学会）以及全球主要 REITs 监管框架列为**必须披露的首要财务指标**。

NOI 的核心价值在于：**剔除资本结构（融资方式）和会计处理差异**，让不同物业之间具有可比性。

### B.2 标准公式体系

行业标准的 NOI 计算遵循自上而下的瀑布结构：

$$NOI = EGI - OpEx$$

$$EGI = PGI - Vacancy\ \&\ Credit\ Loss + Other\ Income$$

各层级含义：

| 层级 | 英文全称 | 含义 | 典型构成 |
|------|---------|------|----------|
| **PGI** | Potential Gross Income | 潜在总收入 | 假设 100% 出租且全额收齐时的总租金收入（含递增后当期租金） |
| **V&CL** | Vacancy & Credit Loss | 空置与坏账损失 | 空置单元的租金损失 + 已租但收不回的坏账 |
| **Other Income** | Other Income | 其他收入 | 停车费、广告位、储藏室、物业管理费转收等非租金经营收入 |
| **EGI** | Effective Gross Income | 有效总收入 | 物业实际有能力获得的总运营收入 |
| **OpEx** | Operating Expenses | 运营支出 | 维持物业正常运转的日常经营开支 |
| **NOI** | Net Operating Income | 净营运收入 | 物业运营产生的净经营现金流 |

### B.3 运营支出（OpEx）行业标准分类

运营支出的分类直接影响 NOI 的准确性。BOMA（国际建筑业主与管理者协会）推荐以下标准分类：

| 支出类目 | 含义 | 本项目对应 |
|---------|------|------------|
| **物业管理费** | 物管人员工资、外包物业服务费 | ✅ 保洁保安（外包物业费） |
| **水电公共能耗** | 公共区域水电、电梯用电、照明 | ✅ 水电公摊 |
| **维修保养** | 日常维修、设备保养（非资本性改良） | ✅ 维修费（工单系统汇入，仅 OpEx 标记部分） |
| **保险费** | 物业综合险、公众责任险 | ✅ 保险 |
| **房产税/物业税** | 从租计征房产税（12%）、土地使用税 | ✅ 税金 |
| **专业服务费** | 法律、审计、消防检测、电梯年检 | ✅ 已纳入运营支出类目 |
| **营销招租费** | 招商佣金、广告宣传 | Phase 2 补充 |
| **行政管理费** | 管理处办公、通讯、差旅 | 可归入物管 |

> **关键原则**：NOI 中的 OpEx 只包含**经常性运营开支**，不包含以下项目：
> - 贷款利息、本金偿还（属融资活动）
> - 折旧与摊销（会计处理，非现金支出）
> - 资本性支出 CapEx（如电梯更换、整层装修）——工单系统中标记为 CapEx 的维修费不计入
> - 所得税（属业主层面税负，非物业运营）
> - 押金收退（代管资金，非收入/支出）

### B.4 管理口径 vs 会计口径

| 维度 | 管理口径（Cash Basis） | 会计口径（Accrual / IFRS 16） |
|------|---------------------|----------------------------|
| **收入确认** | 按合同应收（实际账单金额） | 免租期 → 直线法分摊；递增租金 → 实际利率法 |
| **适用场景** | 日常运营决策、KPI 考核 | 财务报表披露、投资者报告 |
| **优点** | 直观、实时、易理解 | 合规、平滑、符合准则 |
| **缺点** | 免租期月份 NOI 为零（波动大） | 与实际现金流脱节 |
| **PRD 定位** | ✅ Phase 1 采用此口径 | Phase 2 可扩展双口径 |

> 选择管理口径作为 Phase 1 是业界主流做法——绝大多数物业管理系统先上管理口径，会计口径在对接财务系统时再扩展。

### B.5 NOI 衍生指标

| 指标 | 公式 | 用途 |
|------|------|------|
| **Cap Rate（资本化率）** | $Cap\ Rate = \frac{NOI}{物业市场价值}$ | 物业估值/投资回报率评估。Cap Rate 6% 意味着物业价值 ≈ NOI / 0.06 |
| **NOI Margin** | $NOI\ Margin = \frac{NOI}{EGI} \times 100\%$ | 运营效率（已纳入 M3.2 看板），健康值 60%~75% |
| **NOI YoY Growth** | $\frac{NOI_{本期} - NOI_{同期}}{NOI_{同期}} \times 100\%$ | 增长趋势，正增长说明运营改善 |
| **Same-store NOI** | 同一物业组合的 NOI 同比 | 剔除新收购/处置影响的有机增长 |
| **NOI 达成率** | $\frac{实际\ NOI}{预算\ NOI} \times 100\%$ | KPI 考核（对应本系统 K07） |
| **OpEx Ratio** | $\frac{OpEx}{EGI} \times 100\%$ | 支出控制（已纳入 M3.2 看板），健康值 25%~40% |

### B.6 行业基准值

| 物业类型 | NOI Margin 健康范围 | OpEx Ratio | Cap Rate 范围（中国一线） |
|---------|-------------------|------------|----------------------|
| 甲级写字楼 | 65% ~ 75% | 25% ~ 35% | 4.0% ~ 5.5% |
| 零售/商铺 | 60% ~ 70% | 30% ~ 40% | 4.5% ~ 6.5% |
| 公寓/长租 | 55% ~ 65% | 35% ~ 45% | 3.0% ~ 4.5% |
| 混合业态 | 60% ~ 70% | 28% ~ 38% | 视组合而定 |

> 本项目三业态混合（写字楼 69% + 公寓 27% + 商铺 4%）以写字楼为主，NOI Margin 预估健康范围约 **60%~70%**，OpEx Ratio 约 **28%~38%**。

### B.7 NOI 分析实践要点

| 维度 | 说明 |
|------|------|
| **业态分拆** | 三业态 NOI 必须独立计算（本系统已设计），因为公寓和写字楼的收支结构差异巨大 |
| **月度波动** | 集中供暖月份（冬季）、空调能耗月份（夏季）OpEx 会季节性上升，NOI 相应下降，需看 12 个月滚动均值 |
| **空置损失敏感性** | PGI → EGI 的扣减是 NOI 分析中最敏感的变量；出租率每下降 1%，NOI 下降幅度远超 1%（因为 OpEx 大部分是固定的） |
| **维修费归类** | 日常维修（换灯管、修马桶）→ OpEx；整层改造、设备更换 → CapEx，不进 NOI。边界惯例：单次 > 5,000 元且延长资产使用寿命的列 CapEx |
| **预算对比** | NOI 看板的核心价值不仅在于"看现在"，更在于"与预算比"——即 K07 NOI 达成率 |
| **自上而下验证** | PGI → EGI → NOI 每一层都可单独验证：PGI 应与合同台账总额吻合，V&CL 应与出租率匹配，OpEx 应与费用凭证核对 |
