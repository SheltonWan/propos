## 一、需求分析总览

### 1.1 业务目标与核心价值

PropOS 的核心诉求是将 **637 套房源 + 2 套处于特殊状态单元（共 639 套）** 的三业态混合物业（写字楼 69% / 公寓 27% / 商铺 4%）从"Excel + 微信群"模式迁移至数字化统一管理平台，Phase 1 的关键价值闭环如下：

| 价值维度 | 当前痛点 | Phase 1 交付价值 |
|---------|---------|----------------|
| **资产可视化** | 楼层平面与状态靠纸质 CAD，无实时台账 | 楼栋→楼层→单元三级数字台账 + CAD 热区状态色块实时刷新 |
| **租务主动管理** | 到期 / 逾期无预警，完全靠人工 | 智能预警引擎（90/60/30/7/1 天分节点）+ 合同完整状态机 |
| **财务实时化** | NOI 依赖月底人工汇总，滞后严重 | 账单自动生成 + 实时 NOI 看板（经营口径）+ 收款核销轨迹 |
| **工单线上化** | 微信群口头报修，成本零归口 | 全流程线上化工单 + 维修成本自动归入 NOI 支出 |
| **穿透管理** | 二房东转租数据完全不可见 | 主合同→子租赁双级关联 + 外部填报+内部审核双通道 |
| **KPI 量化考核** | 主观评估，缺乏数据支撑 | 10 项系统指标自动取数 + 权重可配置 + 排名/申诉/导出完整闭环 |

## 二、技术可行性分析

### 2.1 技术选型评估

#### 2.1.1 后端：Dart + Shelf ✅ 可行，注意生态限制

| 维度 | 评估 |
|------|------|
| **语言成熟度** | Dart 语言成熟稳定，Shelf 是 Dart 官方 HTTP 框架，中间件管道模式与 Express/Gin 等框架思路一致，学习曲线平缓 |
| **原生 SQL 策略** | 使用 `postgres` 包直写 SQL，无 ORM 魔法，行级隔离逻辑清晰可控，符合 ARCH 设计约束 |
| **生态局限** | 相较 Node.js/Go/Python 生态，Dart Server 端第三方库较少，但 Phase 1 所需能力（JWT、PostgreSQL、文件读写、定时任务）均有成熟方案 |
| **freezed 代码生成** | `freezed` + `json_serializable` 生成不可变数据类，序列化安全，build_runner 构建步骤需纳入 CI/CD |
| **结论** | **可行**，团队需具备 Dart 开发经验；独立包（`rent_escalation_engine` / `kpi_scorer`）零依赖设计正确，有效隔离核心计算逻辑 |

#### 2.1.2 前端：uni-app 4.x + Vue3 Admin ✅ 可行

| 维度 | 评估 |
|------|------|
| **uni-app 4.x 多端复用** | Vue3 + TypeScript + Pinia，一套代码覆盖 iOS/Android/HarmonyOS Next/微信小程序，条件编译处理平台差异，技术路线主流且有大量商业案例 |
| **wot-design-uni** | 适配 uni-app 的 UI 组件库，API 与 Element Plus 风格接近，降低双端开发认知切换成本 |
| **Vue3 Admin（Element Plus）** | PC 管理后台成熟方案，Element Plus + Pinia + Vue Router 4 生态完善，开发效率高 |
| **HarmonyOS Next 支持** | uni-app 4.x 已支持 HarmonyOS Next 编译，但 HarmonyOS Next 生态仍在成熟中，建议对 HarmonyOS 端降低验收优先级，首批上线以 iOS/Android/微信小程序为主 |
| **结论** | **可行**，但 HarmonyOS Next 端建议作为"尽力而为"而非验收强卡口 |

#### 2.1.3 数据库：PostgreSQL 15+ ✅ 可行，设计合理

| 维度 | 评估 |
|------|------|
| **行级安全** | 通过 Repository 层显式 `WHERE master_contract_id = $scope` 实现行级隔离，比 PostgreSQL RLS 更可控、可测试 |
| **JSONB + GIN 索引** | `units.ext_fields` 用 JSONB 存三业态差异字段，GIN 索引支持高效过滤，适合当前业态扩展场景 |
| **TIMESTAMPTZ 全局 UTC** | 统一 UTC 存储，前端 `dayjs` 转换展示，避免时区 Bug，架构约束正确 |
| **M:N 中间表** | `contract_units` 中间表含 `billing_area`/`unit_price`，WALE 计算按单元拆分完全可支撑 |
| **结论** | **可行**，Schema 设计合理，迁移脚本版本化管理规范 |

### 2.2 核心算法可行性

#### 2.2.1 WALE 双口径计算 ✅ 可行

$$WALE_{\text{收入}} = \frac{\sum_i (\max(0,\frac{end\_date_i - NOW()}{365.25\ 天}) \times annual\_rent_i)}{\sum_i annual\_rent_i}$$

- **精确到天**：使用 PostgreSQL `EXTRACT(EPOCH FROM (end_date - NOW())) / 86400.0 / 365.25` 精确计算剩余年份，`GREATEST(..., 0)` 排除已过期合同
- **多单元合同拆分**：JOIN `contract_units` 表，每个 `(contract_id, unit_id)` 行独立计入，避免重复加权
- **提前终止处理**：`termination_type IS NOT NULL` 时 `remaining_years = 0` 自动排除，逻辑清晰
- **计算位置**：封装在后端 `WaleService`，SQL 直接计算聚合结果返回，无需在内存中遍历 639 条记录
- **结论**：算法逻辑明确，SQL 实现无技术障碍，**可行**

#### 2.2.2 租金递增计算引擎 ✅ 可行（需充分测试）

6 种递增类型 + 混合分段的组合是本系统最复杂的计算逻辑，已抽离为独立 package `rent_escalation_engine`：

| 递增类型 | 实现方式 | 复杂度 |
|---------|---------|-------|
| 固定比例 / 固定金额 | 简单乘法 / 加法，基于账期日期 | 低 |
| 阶梯式 | 按租赁年份查表映射单价 | 低 |
| CPI 挂钩 | 查 `cpi_data` 表获取对应年份 CPI，乘以上期租金 | 中（依赖 CPI 数据录入）|
| 每 N 年递增 | 计算账期距起租日的完整 N 年区间，确定当前递增次数 | 中 |
| 免租后基准 | 判断账期是否在免租期内，免租期后以基准价起算 | 中 |
| **混合分段** | 按 `phase` 区间匹配当前账期，分发到对应递增类型计算器 | **高**（多阶段拼接，边界准确性关键）|

- **关键风险**：混合分段在账期恰好跨越阶段边界时（如合同第 2 年最后一天到第 3 年第一天），需精确处理阶段切换，避免少算或重算
- **缓解措施**：`rent_escalation_engine` 要求 100% 单元测试覆盖边界值，必须在 CI 中强制通过
- **结论**：**可行**，但测试覆盖是交付质量保障的关键

#### 2.2.3 KPI 线性插值打分 ✅ 可行

$$Score_i = \begin{cases} 100 & actual \geq fullThreshold\ (\text{正向}) \\ 60 + \frac{actual - passThreshold}{fullThreshold - passThreshold} \times 40 & passThreshold \leq actual < fullThreshold \\ \frac{actual}{passThreshold} \times 60 & actual < passThreshold \end{cases}$$

反向指标（K03/K05/K06/K08）逻辑翻转，已在 ARCH v1.4 中用代码示例明确说明。

- `kpi_scorer` 独立 package，零依赖，单元测试覆盖满分/及格/零分/边界值/反向指标翻转
- K10 租户满意度为手工录入，无自动计算逻辑，存储简单
- **结论**：算法清晰，**可行**

#### 2.2.4 NOI 实时计算 ✅ 可行（口径需明确）

$$NOI = EGI - OpEx = (PGI - VacancyLoss + OtherIncome) - OpEx$$

- **经营管理口径（Phase 1）**：PGI 按合同应收，VacancyLoss 按空置单元参考市场租金估算（依赖运营维护的"参考市场租金"字段），OpEx 含工单维修费 + 物业费 + 水电公摊等
- **不含税口径**：合同必须标注 `tax_inclusive` 字段，账单同时展示含税/不含税，NOI 取不含税值
- **实时性**：账单生成、收款核销、工单完工后触发增量计算，无需全量重算
- **结论**：**可行**，口径声明已明确，无需会计准则调整

### 2.4 CAD 转换链路可行性

PRD 和 ARCH 定义的 CAD 转换链路为：

```
.dwg 文件
  → [ODA File Converter] → .dxf 文件
  → [ezdxf[draw] Python] → SVG / PNG
  → 上传文件存储（S3 兼容或本地）
  → 楼层热区叠加渲染
```

| 环节 | 可行性评估 | 风险点 |
|------|-----------|-------|
| ODA File Converter | ✅ 成熟商业工具，支持批量命令行转换，免费授权用于非商业用途 | ODA 授权条款需核查是否适用于内部商业运营系统 |
| ezdxf[draw] Python | ✅ 开源库，主动维护，SVG/PNG 输出稳定 | 复杂 DWG 图层（如三维图层、特殊 AutoCAD 线型）可能渲染失真 |
| 热区绑定 | ⚠️ 需人工配置 | SVG 层级中识别单元边界需要人工标注或半自动提取，若 CAD 文件图层命名规范统一则可自动化，否则需人工操作 |
| 转换工具运行环境 | ✅ Python `ezdxf` 已纳入 `.venv` 环境 | 需在部署环境中安装 ODA File Converter，Dockerfile 需包含该依赖 |

**关键风险**：热区与 `unit_id` 的精确绑定是 M1 的核心验收项（抽样 30 个单元正确率 100%），但 SVG 中自动识别单元多边形边界依赖 CAD 文件的图层组织质量。若现有 .dwg 文件图层结构不规范，该环节可能成为人力瓶颈。

**建议**：
1. 在项目启动阶段立即对现有 .dwg 文件做试转换评估，识别图层结构质量
2. 若图层不规范，采用"SVG 背景图 + 前端 Canvas 手工绘制热区多边形"的备选方案，热区坐标存储在数据库，不依赖 SVG 自动提取

### 2.5 安全与合规可行性

#### OWASP Top 10 与 PIPL 合规

| 安全域 | 实现方案 | 可行性 |
|-------|---------|--------|
| 身份认证 | JWT（dart_jsonwebtoken），Claims 含 `role` + `bound_contract_id` | ✅ |
| 授权（RBAC） | `rbac_middleware.dart` 路由级守卫，`rolePermissions` 映射表 | ✅ |
| 行级数据隔离 | Repository 层强制 `WHERE master_contract_id = $scope`，不依赖调用方传参 | ✅ 设计正确 |
| 密码安全 | 8 位以上含大小写字母+数字，首次登录强制改密，bcrypt/Argon2 哈希存储 | ✅ |
| 暴力破解防护 | 5 次失败锁定 30 分钟，登录接口限流 10 req/min/IP | ✅ |
| 会话安全 | 30 分钟无操作失效，`session_version` 字段使旧 JWT 失效 | ✅ |
| 传输安全 | 强制 HTTPS/TLS 1.2+（外部门户），内部服务 HTTP 可接受 | ✅ |
| 敏感字段加密 | AES-256-GCM 加密证件号/手机号，API 层默认脱敏（后4位）| ✅ |
| 脱敏还原审计 | 查看完整证件号需特殊权限（`tenantsViewFullCert`），写入审计日志 | ✅ 符合 PIPL §82 |
| SQL 注入防护 | 全部使用参数化查询（`postgres` 包 `$n` 占位符），禁止字符串拼接 SQL | ✅ |
| 数据保留周期 | `data_retention_until` 字段，合同终止后个人信息 ≤ 3 年（PIPL 合规）| ✅ |
| IDOR 防护 | 二房东 ID 不可从路径参数推断其他租户数据，行级隔离在 Repository 强制 | ✅ |

**结论**：安全体系设计完整，可行性高。需在 CI/CD 中加入静态分析（如 `dart analyze`）检查防止参数化查询被绕过。

#### 外部门户 HTTPS 要求

二房东外部填报门户"强制 HTTPS/TLS 1.2+"，需在部署层配置反向代理（Nginx/Caddy）强制 SSL 终止，属于基础设施要求而非代码层，**完全可行**。

## 四、工期影响与分期建议

### 工期影响因子

基于上述可行性分析，以下因子对工期估算有实质影响：

| 因子 | 工期影响 | 说明 |
|------|---------|------|
| CAD 热区绑定（R-01）| +2~4 周 | 若图层不规范，人工标注工作量不可预测 |
| 混合递增规则测试（R-02）| +1 周 | 测试用例设计与边界验证需专项时间 |
| 二房东门户方案确认（GAP-01/R-04）| +0.5 周 | 方案确认后即可启动，影响可控 |
| KPI 正式考核全链路（Must 级）| +2~3 周 | 10 项指标 + 申诉 + 排名 + 导出是一个完整子系统，复杂度被低估的风险最高 |
| `freezed` 代码生成学习成本 | +0.5 周 | 若团队首次使用，需适应 build_runner 工作流 |

### 分期建议（Phase 1 内部优先顺序）

若工期与资源发生冲突，建议按以下优先级顺序交付：

```
Wave 1（核心数据与基础财务闭环）
├── M1: 资产台账（Excel 导入 + 基本列表，暂不含 CAD 热区）
├── M2: 合同录入 + 状态机 + 简单递增规则（固定比例/固定金额/阶梯）
└── M3: 账单生成 + 收款核销 + 基础 NOI（不含 KPI）

Wave 2（可视化与复杂逻辑）
├── M1: CAD 热区导入与颜色渲染
├── M2: 混合递增规则 + WALE 双口径 + 预警引擎
├── M4: 工单全闭环（移动端报修 + PC 派单 + 成本归口）
└── M3: KPI 正式考核（含申诉 + 导出）

Wave 3（穿透与增强）
├── M5: 二房东穿透（填报门户 + 审核 + 穿透看板）
└── Should 项: WALE 趋势图、到期瀑布图、催收模板
```

> **说明**: 此顺序仅是资源紧张时的降级策略。PRD 中 M5 与 KPI 均属 Must 级，若资源允许，Wave 1~3 应并行而非串行。

## 五、结论与决策建议

### 整体可行性结论

**PropOS Phase 1 在技术层面完全可行。** 所选技术栈（Dart/Shelf + uni-app 4.x + Vue3 Admin + PostgreSQL 15+）均为成熟、可落地的方案，无不可解决的技术障碍。系统规模（639 套，50 并发）属于中小型系统，性能风险极低。

### 需要团队立即决策的事项

| 优先级 | 事项 | 建议结论 |
|-------|------|---------|
| P0 | 二房东外部门户前端方案（GAP-01）| 采用 `admin/` 受限路由方案，避免第三个前端工程 |
| P0 | CAD 文件图层质量评估（R-01）| 启动前 2 周完成 3~5 个楼层试转换，判断热区自动提取可行性 |
| P1 | ODA File Converter 授权确认（R-07）| 与 ODA Forum 确认授权或切换为开源替代 |
| P1 | NOI 预算录入入口补充（GAP-02）| 在 KPI 方案配置中补充"预算 NOI"字段 |
| P2 | HarmonyOS 端验收标准降级 | 明确 HarmonyOS 端作为"尽力而为"，不作为首批上线强卡口 |

### 需求质量评价

PRD v1.8 的需求定义质量较高，大多数边界场景、异常流程、权限约束均有明确说明，与 ARCH v1.4 的架构设计对齐良好。文档中隐含需求较少，仅 GAP-01 至 GAP-05 共 5 项需在实现阶段补充澄清，总体需求完整性评分：**8.5 / 10**。
