# 智慧物业资产运营管理平台 PRD 修改意见

> 适用文档: PRD v1.5
> 评审日期: 2026-04-06
> 评审目标: 按行业常见的商业地产资管/租务/物业数字化项目标准，对当前需求进行范围、规则、风控与验收可执行性把关，并形成 v1.6 修订方向。

---

## 一、评审结论

当前 PRD v1.5 的业务覆盖面完整，已经具备 Phase 1 方案设计基础，但仍存在以下共性问题：

1. 首版范围偏大，核心闭环与增强能力混杂，存在排期失真与交付失控风险。
2. 财务计费、核销、押金、发票等规则定义不够细，后续容易引发财务口径争议。
3. 二房东外部门户涉及敏感数据采集和外部访问，但安全与合规控制仍偏粗。
4. 预警、工单、审核等流程只有主流程，缺少 SLA、异常流和责任归属定义。
5. 验收标准偏结果描述，缺少样本、阈值、步骤和责任方，不利于项目验收。

综合判断：

- 该 PRD 适合继续深化，不建议直接作为最终开发基线。
- 建议升级为 v1.6，并补足“范围分级、计费口径、安全矩阵、流程 SLA、验收用例矩阵”五类内容。

---

## 二、高优先级修改意见

### 1. Phase 1 范围必须分级

当前 Phase 1 同时覆盖资产可视化、合同、财务、KPI、移动工单、小程序、二房东 Web 填报、穿透分析看板和 API 预留，首版范围偏大。

建议拆分为三层：

- Must：上线必须完成，直接构成业务闭环。
- Should：有较高价值，但不作为首批上线阻塞项。
- Could：作为 Phase 1.5 或 Phase 2 预留。

建议归类如下：

| 分级 | 建议纳入项 |
|------|-----------|
| Must | 资产台账、CAD 基础展示、合同录入与状态机、递增规则、自动账单、收款核销、基础 NOI、内部工单闭环、二房东人工录入、二房东 Web 填报与审核、基础审计日志 |
| Should | 资产概览看板、WALE 趋势、催收模板、子租赁穿透视图、楼栋/业态下钻报表 |
| Could | KPI 排名与历史趋势、二房东 API 对接、复杂溢价分析、微信小程序增强能力、外部消息通道扩展 |

---

### 2. 财务规则必须从“功能”提升为“口径”

当前 PRD 已写明自动账单与 NOI，但仍缺以下关键口径：

1. 租金计费基准：按建筑面积、套内面积还是固定总价。
2. 首期/末期不足月的按天折算规则。
3. 免租/装修期跨账期时的处理方式。
4. 押金收取、退还、冲抵、冻结条件。
5. 部分收款、拆分收款、跨账单核销规则。
6. 发票状态与收款状态是否解耦。
7. 商铺营业额分成的申报时间、校验责任、补报机制。
8. NOI 是否按含税、未税还是现金口径展示。

建议在 v1.6 中新增独立章节“计费与核销规则矩阵”。

---

### 3. 外部门户的安全与合规要求需要显式化

当前二房东模块已经涉及外部登录、终端租客身份信息、联系方式和实际租金数据，属于高敏感场景。

建议补充以下要求：

1. 密码复杂度、登录失败锁定、首次登录强制改密。
2. 会话超时、异地登录失效、账号冻结规则。
3. 证件号、手机号、联系人信息的查看、导出与审批权限。
4. 审计日志留存周期与可追溯字段。
5. 数据导出权限与水印要求。
6. 二房东填报数据的授权责任说明，由谁承诺真实性与合法性。
7. 主合同到期后账号冻结但数据只读保留的策略。

---

### 4. 流程类需求需要补齐 SLA 和异常流

目前合同预警、工单流转、二房东审核等需求以主流程为主，缺少实际运营必需的异常处理。

建议至少补齐以下内容：

1. 工单派单 SLA、完工 SLA、超时升级规则。
2. 审核退回后是否允许重提、重提后是否保留版本。
3. 通知失败后的补偿与重试机制。
4. 预警触发失败时的巡检与补发机制。
5. 工单完工后是否允许重开、由谁批准。
6. 子租赁审核超时后的自动提醒与升级。

---

### 5. KPI 建议降级为“试运行分析”

KPI 模块业务上有价值，但在首版就进入员工/部门正式评分与排名，前提条件并不充分。

主要缺口包括：

1. 预算 NOI 数据来源未定义。
2. 部门组织架构、员工归属变更规则未定义。
3. 数据冻结时间与补录后是否回算未定义。
4. KPI 争议处理机制未定义。

建议 v1.6 将 KPI 重新定义为：

- Phase 1：经营分析与试运行评分，不直接作为正式绩效依据。
- 后续版本：在组织、预算、归属、冻结机制稳定后再升级为正式考核模块。

---

## 三、中优先级修改意见

### 1. 资产图纸模块需要补版本管理规则

建议增加：

1. 图纸与单元热区的绑定方式。
2. 图纸替换后的版本留存。
3. 单元拆分、合并、停租、不可租后的历史处理。
4. 同楼层多张图纸或多图层的管理方式。

### 2. 数据初始化需要补数据治理要求

建议增加：

1. Excel 模板字段级校验说明。
2. 导入失败回滚规则。
3. 数据主责部门与最终确认人。
4. 历史合同和未结账单的抽样复核比例。

### 3. 非功能要求需要补运维指标

建议增加：

1. RTO / RPO 目标。
2. 关键任务调度失败重试策略。
3. 邮件、短信、推送的失败告警。
4. 日志、审计、文件存储的留存周期。

---

## 四、建议纳入 v1.6 的具体修订项

建议在 PRD v1.6 中新增或修改以下内容：

1. 在 Phase 1 前增加“范围分级与交付边界”章节。
2. 在合同/财务模块中增加“计费与核销规则矩阵”。
3. 在预警、工单、二房东审核处增加 SLA 和异常流说明。
4. 扩展二房东安全与权限设计，增加会话、导出、审批和留存控制。
5. 将 KPI 模块定位调整为“经营分析 + 试运行评分”。
6. 强化非功能要求，补充可观测性、恢复目标和任务补偿策略。
7. 将验收标准改写为“测试场景 + 通过阈值”的形式。

---

## 五、修订后预期收益

完成上述修订后，PRD 将从“功能设想文档”提升为“可执行交付基线”，主要收益包括：

1. 开发范围更清晰，避免首版失控。
2. 财务口径更统一，减少上线后的解释成本。
3. 外部门户风险更可控，降低合规隐患。
4. 流程更可执行，便于运营团队实际落地。
5. 验收更客观，减少项目收口阶段争议。

---

## 六、建议执行顺序

建议按以下顺序推进：

1. 先锁定 Phase 1 Must 范围。
2. 再补齐计费、核销、安全、SLA 规则。
3. 然后同步修订验收标准。
4. 最后将 Should / Could 需求放入后续迭代计划。

---

## 七、结论

PRD v1.5 不需要推翻重写，但必须进行一次面向交付的结构性升级。建议以 v1.6 作为新的需求基线版本，并将“规则、边界、验收”作为本轮修订重点。
